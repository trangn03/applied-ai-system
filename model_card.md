# PawPal+ — Model Card

This markdown documents the AI-assisted layer in `agent.py` (Gemini `gemini-2.5-flash`, wrapped around the deterministic `Scheduler` in `pawpal_system.py`). See `README.md` for the full architecture and `diagrams/uml.mmd` for the system diagram.

## Limitations and Biases

- **Only reads plain text, nothing deeper.** The AI only sees a task's title, start time, duration, and priority as text (`_describe` in `agent.py`) — no knowledge of the pet's actual health history or the owner's real preferences, just whatever's typed into a task title. So when the AI decides "medication looks under-prioritized," that judgment is really just pattern-matching on words like "medicine" or "vet" sounding urgent, not real medical reasoning. A vaguely named task like "Give Buddy his thing" could easily get judged wrong.
- **Looks at one task at a time, not the whole day.** `suggest_priority_overrides` judges each task's priority alone, with no sense of how bumping one task up affects everything else on the schedule. That's exactly why `generate_plan` double-checks the outcome afterward (the Check/Revise step) instead of just trusting the AI's first answer.
- **A small, fast model, not a deep thinker.** `agent.py` uses `gemini-2.5-flash` with reasoning turned off (`thinking_budget=0`) and a short 500-token limit, to keep the app fast. That trade-off works for quick lookups and short summaries, but not for careful, nuanced judgment calls.
- **No memory across runs.** Priority overrides are never saved (`_with_overrides` only changes in-memory copies), so the AI can't learn an owner's actual preferences over time — every "Generate schedule" click starts fresh.
- **Only one fixed format is understood.** The scheduler's three priority levels (`high`/`medium`/`low`) and `"HH:MM"` time format are the only inputs and outputs the AI works with — no other languages, no free-text urgency, no other time formats.
- **Training-data biases can carry over.** The AI can favor whichever words "sound" important (e.g. "medicine" or "vet" vs. "playtime" or "grooming") based on patterns learned during training — this hasn't been checked against actual veterinary guidance, so that instinct is a guess dressed up as judgment.

## Potential Misuse and Mitigations

- **Someone could try to manipulate the AI through a task title.** A task title is user-typed text, and that text gets sent straight into the prompt the AI reads (`_describe`, `suggest_priority_overrides`, `explain_plan`). A crafted title could try to trick the AI into ignoring its instructions. How this is prevented: the AI's answer is never trusted blindly. `suggest_priority_overrides` only accepts a strict list of real task names and one of the three valid priority words (checked in `agent.py` against `known_keys` and `valid = {"high","medium","low"}`); anything else gets thrown out. `explain_plan` discards any summary that doesn't mention a real task. The AI's text is only ever shown on screen or used to set one of those three priority values — never run as code, never used to control the app.
- **Someone could rack up API costs.** Every time the "Use AI" box is checked, that click costs a real API call to Gemini. If this app were put online for anyone to use, someone could spam that button and run up the bill. How this is prevented: right now this is a personal/local project with no login system, so a public deployment isn't the intended use case — a public version would need rate limits or separate API keys per user.
- **The API key itself could leak.** The key lives in an environment variable or `.streamlit/secrets.toml`. If that file were accidentally shared or committed, someone else could use the key and run up charges on the owner's account. How this is prevented: `.streamlit/secrets.toml` is excluded from git, the README tells users to keep the key as a local secret instead of hardcoding it, and a leaked key should just be rotated.
- **A wrong AI summary could mislead the owner.** Since this touches pet care (like medication timing), a made-up or wrong explanation could actually cause harm if trusted. How this is prevented: `explain_plan` fails safe — when the explanation can't be confirmed as grounded in a real task, `explain_plan` raises an error and the app just shows no summary instead of a risky one.

## Surprises From Testing

- **The AI can sound confident while being completely wrong.** While testing `explain_plan`, feeding the model a scenario where the wrong task might get described showed how easily an AI can write a smooth, convincing explanation about something that never actually happened. That's exactly why every response has to be checked against the real task list before anyone sees it.
- **Formatting instructions aren't always followed.** Even when told directly "respond with JSON only, no extra text," the AI sometimes still doesn't comply. `suggest_priority_overrides` needs a `try/except` around the JSON parsing just to handle that.
- **A change that sounds reasonable can still make things worse overall.** While building the Check/Revise loop, bumping one task's priority — because that task looked under-prioritized on its own — turned out to sometimes cause a different pet to lose every one of its scheduled tasks. The suggestion sounded fine in isolation but was bad for the bigger picture. That's why the result gets double-checked after the fact instead of trusting the AI's own reasoning for making the change.
- **Silent failures from an unexpected setting.** Early on, some responses came back completely empty. The model's internal "thinking" step turned out to be quietly using up the whole output budget before writing anything visible. Turning that step off (`thinking_config=types.ThinkingConfig(thinking_budget=0)`) fixed the empty-response problem — not an obvious thing to track down.

## AI Collaboration on This Project

I used Claude throughout the design and build of this AI layer — for brainstorming the Plan → Act → Check → Revise → Explain structure, writing tests, and debugging.

**A helpful suggestion.** While designing `generate_plan`, Claude suggested not trusting a priority change just because the AI proposed the change confidently. Instead, re-run the real `Scheduler` with the change applied and check whether that change caused a pet that used to get a task to end up with none (`_starved_pets`) — and if so, undo the change and fall back to the plain scheduler. This Check/Revise safety net is exactly what `test_generate_plan_reverts_when_override_cannot_avoid_starvation` checks, and this safety net is the reason a bad AI suggestion can never make the actual schedule worse than the plain, non-AI version.

**A flawed suggestion.** An early version of `suggest_priority_overrides` just parsed the AI's JSON response and used the result directly, assuming the instructions given to the AI would be enough to keep the response well-formed. That assumption was wrong two ways: the AI could name a task that doesn't exist or a priority word outside `high`/`medium`/`low`, and the AI didn't always return valid JSON at all. I didn't accept that as good enough — the fix was to filter the response down to only real task names and only the three valid priority words, and to catch parsing errors and fall back to "no changes" whenever the response looked off. `test_suggest_priority_overrides_filters_unknown_keys_and_invalid_priorities` and `test_suggest_priority_overrides_returns_empty_on_malformed_json` lock that fix in.
